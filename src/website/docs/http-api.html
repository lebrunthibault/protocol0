HTTP API

The remote script inside Ableton runs a tiny localhost HTTP server — a real REST API under /api. It's how the agent triggers actions on a keypress, and it's open for you to call from a script or any tool that can issue HTTP requests. It even ships its own interactive Swagger UI.

Base URL

http://127.0.0.1:9000
  • Localhost only. The server binds to 127.0.0.1, so it's never reachable from the network — your data stays on the machine.
  • Port 9000 by default, but dynamic. If 9000 is taken, the script picks another free port and publishes the effective one in %APPDATA%\Protocol0\runtime.json. Read that file if you need the exact URL.
  • REST under /api. Reads are GET; mutations are POST with a JSON body. (Earlier versions exposed everything as GET at the root — that has moved.)
  • Available whenever Ableton is open with the Protocol 0 control surface enabled — it lives and dies with Ableton.

Open http://127.0.0.1:9000/docs in a browser for an interactive Swagger UI listing every endpoint, its parameters, and its description — generated live from the running script (the machine-readable spec is at /openapi.json). Use the port from runtime.json if the script isn't on 9000. Visiting / redirects there.

Action endpoints

These drive Live directly. They're the same calls a keyboard shortcut makes under the hood. Mutations are POST with a JSON body.

Load a device

POST /api/device/load     { "name": "<str>" }

Loads a device (instrument or audio effect) by name onto the selected track.

curl -X POST http://127.0.0.1:9000/api/device/load \
     -H "Content-Type: application/json" \
     -d '{"name": "EQ Eight"}'

Select a track

POST /api/track/select    { "name": "<str>" }

Selects a track by name. Pass "name": "master" to select the master track.

Stop following the playhead

POST /api/song/toggle_follow

Stops Live from following the playhead in the arrangement view.

Read the set state

GET /api/set/get_state

Returns the full serialized state of the current Ableton set as JSON. This one is a read, so it's a GET.

Some routes exist but aren't yet exposed in the shortcut catalog the web UI shows. They're still callable directly over HTTP. The set of routes grows over time — the live Swagger UI at /docs is always the source of truth.

Managing bindings

Shortcut bindings are not managed through the script. They're owned by the agent, which serves the configuration web UI and a small /api on http://127.0.0.1:9010 — separate from the action API documented here. The agent stays up independently of Ableton, so you can edit bindings even with Live closed. To manage shortcuts, use the web UI; to drive Live directly, call the action endpoints above.

Example: driving it from external gear

Any tool that can send an HTTP POST can trigger an action. For a Stream Deck, use a plugin that issues an HTTP request (a plain "open URL" button only does GET, which no longer triggers mutations):

POST http://127.0.0.1:9000/api/device/load
Content-Type: application/json

{"name": "Utility"}

That loads a Utility device onto the selected track — no keyboard shortcut required.
