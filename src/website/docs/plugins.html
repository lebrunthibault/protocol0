Extending

Creating plugins

The remote script that runs inside Ableton is extensible through small plugins. A plugin can react to things happening in Live — a clip recorded, playback started, a track selected — and expose new actions that you can bind to a shortcut. You add behavior without touching the core.

New to the Live control-surface framework that Protocol 0 builds on? Structure Void's architecture guide is the best primer on how a MIDI Remote Script talks to Live.

This is a developer topic — it assumes you run Protocol 0 from source. The full reference, with a copy-paste template, lives in docs/plugins.md.

What a plugin does

  • React to events. Declare a map of event → handler; the script subscribes and unsubscribes for you.
  • Expose actions. An @api_route function becomes an HTTP endpoint under /api, shown in the Swagger UI and callable from a shortcut or external tool.

Where plugins live

Plugins are Python packages under src/script/protocol0/plugins/. They're discovered automatically at startup — subclass PluginInterface and drop the file in; there is no registry to edit.

src/script/protocol0/plugins/
  example/      ← copy this to start your own
  live_set/     ← the author's real-world plugin

The interface

Everything except name is optional. A purely declarative plugin only implements register_listeners and/or register_actions.

class PluginInterface:
    name: str = ""

    def should_start(self) -> bool: ...        # run this plugin? (default: True)
    def start(self) -> None: ...               # optional one-time setup
    def stop(self) -> None: ...                # optional teardown
    def register_listeners(self): ...          # {EventType: handler}
    def register_actions(self): ...            # [route functions]

Reacting to events

from protocol0.domain.lom.clip.ClipRecordedEvent import ClipRecordedEvent

class MyPlugin(PluginInterface):
    name = "my_plugin"

    def register_listeners(self):
        return {ClipRecordedEvent: self._on_clip_recorded}

    def _on_clip_recorded(self, event: ClipRecordedEvent):
        clip = event.clip            # events carry their payload as attributes
        ...

A few commonly useful events:

Adding an action

An action is an @api_route function — exactly like the built-in ones (/api/device/load, …). It shows up in the Swagger UI at /docs and is callable over HTTP. Mutations are POST (arguments in a JSON body); reads are GET.

from protocol0.application.http.Router import api_route

@api_route("POST", "/my_plugin/do_thing")  # registers /api/my_plugin/do_thing
def do_thing(name: str) -> None:
    """Short docstring — shown as the action's summary in the Swagger UI."""
    ...

class MyPlugin(PluginInterface):
    name = "my_plugin"

    def register_actions(self):
        return [do_thing]

Arguments are coerced from the signature (name: str, count: int, flag: bool) — from the query string for GET, from the JSON body for POST; the return type drives the response (nothing → fire-and-forget, dict/list → JSON). The handler runs on Live's thread, so calling the Live API from it is safe.

Constraints

  • Stdlib-only. Ableton's embedded Python can't load third-party packages.
  • Never block Live's thread. Keep listeners and actions short.
  • One failing plugin won't take the script down — the loader isolates each plugin's startup.

Per-plugin configuration isn't available yet; it's tracked on the project's spec backlog. For now keep plugin behavior self-contained.
